# 评标工作台设计

更新时间：2026-07-21

## 目标与范围

新增独立 Web 模块“评标工作台”，用于单人、低并发的评标辅助。入口为 `/pingbiao`，API 前缀为 `/api/evaluation-workbench`。

模块包含：

- 评标项目、招标文件/附件与多份投标文件管理；
- 多投标文件查重与横向结果汇总；
- 招标规则提取、人工确认，以及资格性、符合性、实质性和废标点审查；
- 客观分的证据提取与程序计分；
- 主观分的证据化辅助审查；
- 浏览器打印与 CSV 评分报告导出；XLSX、DOCX 留作后续兼容性扩展。

明确不包含：投标文件或技术方案编写、AI 生图、OCR、DOC/WPS 转换、本地大模型、多用户协作，以及此前单独列出的上线安全整改。

## 独立边界

- 不修改 `/bijiao`、`/api/compare` 及其响应结构；
- 不影响其他 Blueprint 或 `rokid_glasses_app` 调用的 API；
- 使用独立的 SQLite 数据库 `data/evaluation_workspace.db`；
- 原始文件与解析缓存存储在 `data/evaluation_workspace/<project_id>/`，结构化结果存储在独立 SQLite；打印报告按需生成，不常驻占用磁盘；
- 新功能使用独立 Blueprint、模板、静态资源、服务与任务进程；
- 只在导航页增加“评标工作台”入口。

## 文件与资源限制

- 第一版支持 PDF 和 DOCX；不做扫描件 OCR，未提取到可检索文本的扫描件会明确标记解析失败；
- 单文件上限 100 MB，PDF 解析上限 2000 页，可提取文本上限 200 万字符；
- 每项目默认最多 10 份投标文件；
- 单文件和整批页数、字符数沿用并逐步复用现有 `/bijiao` 的资源边界；
- 文件解析、文件对比和 AI 请求默认并发均为 1；
- 最多保留 3 个排队任务；
- 评标工作台空闲时不保留工作进程、不加载模型、不建立 AI 连接，也不执行定时扫描。

## 工作流

1. 输入新建项目口令后，创建项目并上传一份主招标文件、招标附件和多份投标文件。
2. 按顺序解析文件并按文件哈希缓存结果。
3. 多投标文件顺序两两比较；本地程序先提取低成本候选证据，再由 AI 对压缩后的候选证据包作结论，形成文件关系矩阵和多文件共同线索。
4. 从招标文件提取资格、符合、实质性、废标、客观分和主观分规则。
5. 用户确认、修改和版本化规则。
6. 对每份投标文件执行“综合评审”：系统按审查、客观分、主观分及小规则组调用 AI，结果仍按审查、客观分、主观分分区展示。
7. 汇总横向结果并导出报告。

## 查重结果

查重沿用现有比对模块对 PDF 的高价值规则：完全雷同、近似雷同、已验证招标共同修改、共同错误、敏感实体、元数据辅助线索及招标原文排除。在此之上增加“横向异常线索”层，分别记录线索维度、S3 人工核验等级、可信度、证据状态、反证提示、判定结果和人工处置，不把多个字段压缩成一个不透明的风险分。

本地程序仅用于生成可比对的候选证据和压缩证据包，不直接给出查重结论。证据包包含投标人名称、各自页码（如“甲公司第 7 页 / 乙公司第 4 页”）、规则名称、最多三条短原文、相似度/次数及反证提示。AI 按批复核这些证据包，输出“确认线索、疑似线索、倾向排除或证据不足”、风险、置信度和建议核验事项；不发送完整投标文件对，不按文件对逐次调用。没有候选证据不调用模型，AI 故障时仅标记 AI 判定未完成。查重复用键包含文件哈希、规则/提示版本和模型公开配置，未变化时自动复用完成结果。

系统仅生成 `collusion_signal`（串标相关线索），不自动生成或认定 `statutory_collusion_condition`（法定串标情形）。多维命中只提高人工复核优先级，不提高法律定性，不触发废标或扣分；工作台不保存“人工处置”状态。

当前实际执行正文雷同、共同异常或错误、共同联系电话、共同邮箱、共同姓名、共同身份证信息、共同地址、招标原文共同改动和相同文档属性九个维度。姓名和地址仅从带明确字段标签的内容中提取，以减少普通文本或模板造成的误报；联系方式、邮箱、姓名、身份证信息和地址在仅限评标使用的横向汇总中完整展示，以便人工核对；元数据只作辅助线索。报价规律、缴费来源、业绩引用、他方主体残留和共同理解偏差因缺少可靠结构化来源而明确披露为“未执行”，禁止从正文数字或模型猜测补齐。后续接入结构化报价时，应使用 Decimal 和中位数/MAD/IQR 等稳健统计，小于 4 个有效样本只做描述性展示。

公共模板处理遵循“先排除、再比较”：提供主招标文件时排除招标原文直接复制，并保留共同实质改动；未提供招标文件时明确降低结果解释力度。其他公共行业模板若未获得确认，不得把模板相似直接升级为串标结论。

结果包含：

- 文件关系矩阵：每个文件对的独立线索维度、线索数量和人工复核优先级；
- 正文雷同、目录、元数据、图片、共同错误、共同修改；
- 文件标签、页码、原文对照和风险说明。

图片第一版只做原始图片哈希与位置记录，不做 AI 图像语义识别。

## 审查与评分

招标规则须先由 AI 提取并经用户确认。规则提取提示词要求模型基于完整语境排除仅能在递交、密封、开标现场或纸质份数环节核验的要求；后端不再以关键词硬过滤，避免把“投标截止日前若干年”等业绩有效期误删。每条规则同时保留“检查规则”（面向 AI 和页面的明确核验指令）与“招标原文依据”（可追溯的原文摘录）、名称、类型、页码、是否需要 OCR、评分方式和启用状态；所有保留的检查点均交由 AI 判断。页面以检查规则为主，原文依据用于核验与报告追溯；规则修改后形成新版本，旧结果保留并标记其使用的规则版本。人工补充规则可增加资格性、符合性、实质性/废标项、客观分、主观分和其他规则；补充客观分或主观分时必须明确满分，客观分还可选择“满足即满分”或“按评分规则由 AI 建议得分”。当前规则集不展示具体评分参数，保持规则浏览界面简洁。

规则提取会先在本地定位“得 X 分、最高 X 分、满分 X 分”等疑似评分条款，并把短片段作为 AI 的评分覆盖清单。系统按每条评分片段与已提取评分规则的名称进行本地覆盖校验，不能再以评分规则总数替代逐条校验；仅把未覆盖的短片段和既有评分规则发送给 AI 进行一次遗漏补提取，再合并去重。本地不会直接生成评分规则或分数。这一机制适用于业绩、报价、人员、资质、方案及其他明确计分项。

当前规则集在待确认状态下默认启用全部规则。用户可逐条取消“启用”；未启用的规则会保留在该版本中以便追溯，但不参与确认后的综合评审或评分。页面不提供删除规则入口，以启用状态作为规则取舍的唯一操作；至少保留一条启用规则后方可确认规则集。为兼容既有调用，后端删除接口暂时保留但不在页面暴露。

资格、符合、实质性和废标审查结果统一使用：满足、不满足、部分满足、未找到证据、需 OCR 后判定、需人工判断。页面以中文展示 AI 结论，并按投标人分组，在每位投标人内按照异常优先、高中低风险排序；分组可以展开或收起。每条结果包含招标依据、投标证据、页码、理由、风险级别、置信度和证据完整度。

工作台采用 AI 优先，页面仅展示 AI 建议，不提供一键确认、人工调整、有效分或最终分写回。评分页面、打印报告和 CSV 都明确展示“AI 建议得分”。需要图片文字或盖章识别的规则会被明确标为“需 OCR 后判定/评分”；当前版本没有 OCR 引擎，不能据此把投标文件判为不满足。

客观分与主观分均由 AI 根据明确的招标依据给出建议分、证据页码和理由；不保存人工调整、有效分或最终分。数量、阈值、分档、累计上限和价格公式仍只以受限规则结构扩展，禁止执行模型返回的表达式。

AI 提取客观分和主观分时必须返回 `scoring.max_score`。若模型遗漏，但招标原文明确包含“总计/最高/最多/满分 X 分”等表述，工作台仅从该原文补齐满分，并标记为“从招标依据识别”；对于数量累计、区间、分档等客观分，默认保持人工评分，不会误作“满足即满分”。待确认规则中的评分满分可直接在规则表内修改保存，无需删除规则重建。

主观分先按同一规则版本、模型档案和 Prompt 版本独立审查每份投标文件，再做横向汇总。客观分与主观分结果均按投标人分组展示。输出 AI 建议得分、证据页码、优点、不足、得扣分理由和置信度。

## 模型档案

模型档案是工作台级全局配置，通过页面右上角“模型配置”入口维护，不归属于任何单一评标项目。项目内各任务只选择已配置的模型档案。

模型档案保存显示名、协议、Base URL、模型名、上下文长度、超时、JSON 模式和思考模式。API Key 可以手动录入：系统用工作台独立密钥加密后保存，密钥文件位于被忽略的 `data/` 持久化目录；读取、编辑、模型列表和 API 响应均不返回明文或密文，只显示“已配置”。旧的 API Key 环境变量配置继续兼容。

每个模型档案提供“测试连接”，以极小请求验证 Base URL、模型 ID、API Key 和兼容参数。测试不写入评标项目数据，也不记录 API Key。

## 提示词配置

页面右上角提供与模型配置同级的“提示词配置”入口，使用相同的 `108` 配置口令。可查看、修改或恢复默认的模型连接测试、文件查重 AI 复核、评审规则提取与评分补充、单项审查、客观分、主观分、综合评审全文覆盖扫描和最终规则组复核的系统提示词及任务提示词；修改作用于后续新任务，且会进入任务复用指纹，避免错误复用旧结果。规则、文件正文、证据包及 JSON 输出所需的运行时数据以 `{{变量名}}` 占位符注入；保存时会校验必需变量未被删除，因此用户可以编辑全部指令文本而不会丢失运行时数据。

## 全局通用规则库

页面右上角提供与模型配置、提示词配置同级的“通用规则库”，可直接查看，无需口令；新增、修改或删除每次都必须输入 `108`，服务端逐次校验。规则库仅支持资格性、符合性、实质性/废标项和其他规则，每项包括名称、明确的检查规则、可选招标原文依据及 OCR 标识。启用的规则会在**新项目创建**以及每次**重新提取待确认规则**时自动复制到该项目的待确认规则集；不提供手动选择或导入，以保持每个新项目的基线一致。提取时，和 AI 规则在分类、规则名称、检查规则三项完全一致的通用规则仅保留一条，避免重复。规则库的后续新增、修改、停用或删除不会回写既有项目，项目内仍可在确认前按实际招标文件编辑或删除已复制规则。

“重新提取规则”始终强制启动新的 AI 提取任务，不复用此前任务结果；已有待确认规则时会先在页面提示，新规则集将替换当前待确认规则集。被替换的规则版本、已确认版本及历史审查结果仍保留，不会删除。

可将已启用且已配置 API Key 的模型设为全局默认模型。新建任务和页面初始选择器自动采用该默认模型，用户仍可为某一次任务临时改选；历史任务继续保存实际使用的档案 ID。删除默认模型时，系统会自动改用另一个已启用且已配置 API Key 的模型；没有可用替代项则清除默认设置。若模型正被排队或运行中的任务使用，系统仍拒绝删除。

内置 DeepSeek 预设：

- `deepseek-v4-flash`：规则提取、结构识别、错别字、客观分证据提取；
- `deepseek-v4-pro`：复杂实质性/废标审查、主观分与横向总结；
- Base URL：`https://api.deepseek.com`。

其他模型通过 OpenAI-compatible 适配器接入；每次任务可在已配置档案中选择模型。初始迁移会将已配置的 `deepseek-v4-flash` 设为全局默认模型；以后所有页面与服务端均优先使用管理员明确设置的全局默认模型，再回退至第一个可用模型。

## 准确性优先的全文覆盖与上下文控制

页面只保留“综合评审”入口。评审基线是准确性优先于 Token 节省：超过 2.4 万字符的投标文件先按页顺序切成约 1.1 万字符、相邻页有重叠的连续块，AI 必须从第一页覆盖到最后一页。全文首轮采用“证据索引”而非完整评审：每个页块只发送一次全部启用规则的精简目录，返回 `规则 ID、页码、短原文、支持/反向/疑点倾向` 四项候选；不在首轮生成理由、建议分、数量、OCR 结论或置信度。较短文件直接把全文交给最终规则组。随后系统按审查、客观分、主观分小组汇总候选、补充重点章节原文并只在最终阶段形成结论、逐项业绩、计分过程和建议分。点状短语、章节标题和本地实体匹配只用于加强二次复核，不再决定某一页是否会被 AI 看到。

所有 AI 结论均由人工最终审核。提示词要求模型积极给出最可能的结论、风险、疑点、证据、理由和建议分；证据不完整或需要 OCR 时也可给出暂定判断和暂定建议分，并明确核验点。系统不再因为首轮没有候选而默认转为“需人工全文通读”，而会说明已完成全文扫描并给出“未发现相应异常/材料”或其他最可能建议。积极判断不等于允许虚构：模型不得编造原文、页码、签章、图片或线下材料，必须区分可见事实与 AI 推断。

系统为每次模型调用记录任务、阶段、模型、发送字符数、供应商返回的输入/输出/总 Token、缓存命中 Token 和上下文模式；对于规则提取和综合评审还记录请求的输出上限、服务商结束原因和响应字符数，用于定位截断，不记录提示词正文、投标文件原文、模型思考内容或 API Key。页面仅显示项目汇总。对于不返回 usage 的 OpenAI-compatible 模型，页面显示发送字符数。

全文扫描采用按需运行的工作进程，不使用向量数据库、常驻索引或本地模型。进程同一时间只发送一个模型请求，内存中只保留当前文件的文本块和短候选，任务结束即释放；不运行评标工作台时不会增加常驻 CPU 或模型资源。每个成功页块的紧凑候选会以“文件、模型、精简规则目录、提示词版本、页块哈希”作为键立即写入工作台 SQLite；工作进程中断或强制重新运行时会跳过相同的已完成页块。全文覆盖仍比旧的候选页模式耗时，但不会再形成“页块 × 规则批次”的重复扫描矩阵。

全文扫描的单个页块若返回坏 JSON，会先在不永久改变模型档案的情况下临时禁用思考并严格重试；仍失败时只把精简规则目录拆为两组，绝不递归拆分页块并重发全文。无法恢复的极少数页块不会静默丢弃，其原文会直接附入相关最终规则组二次复核。最终规则组仍按审查和客观分最多 8 条、主观分最多 6 条执行，完成一组即持久化，单条规则最终格式异常不覆盖此前成功结果。

数量、累计和分档客观分必须尽量输出 `matched_count`、逐项证据、页码、有效性、计分过程和 `suggested_score`。后端优先采用模型在满分范围内给出的建议分；若模型已返回数量且规则含“每项/每个得 X 分”，后端可按明确公式补算并限制在满分以内。业绩结果会把数量和逐项证据整理进现有证据、理由字段，使页面、报告和 CSV 均能显示“AI 识别几项、哪些建议有效、哪些待 OCR、如何得到建议分”。需要 OCR 时暂定建议分仍保留，但不会进入自动确认状态。

规则提取采用“原文映射 → 规则编译 → 覆盖审计”三阶段提示词链路，而非依赖地区、设备或评分关键词的业务补丁。系统先按页或段落拆成约 1.1 万字符的独立批次，逐段映射原始条款、原文依据和页码；当候选规则较多时，再由 AI 统一合并语义重复项、保留评分公式与上限，并以覆盖审计补回确实遗漏的可核验条款。评分正则仅作为低成本提醒，不能单独决定规则完整性。MiniMax M3 等支持该参数的模型在严格 JSON 阶段临时禁用思考模式，避免思考内容占用结构化结论的输出空间；模型档案的默认思考设置不受影响。若单个批次仍因长度截断，系统只将该批二分并重试；只有无法再拆分的小批次才用不降低输出预算的紧凑 JSON 重试。该流程不创建常驻队列、向量库或本地模型，任务结束后释放内存，适配 2 核 2 GB 服务器。

最终规则组按任务类型限制上下文：点状核验约 4.2 万字符、全文一致性约 5.5 万字符、数量/业绩和方案章节约 6.4 万字符，防止二次汇总本身超过模型上下文；数量/业绩类规则优先携带更多候选和章节页，方案类规则优先携带连续方案页，审查规则组同时携带优先级最高的项目范围偏离候选及原页。全文扫描会为每条证据标注其对最终判断的直接性；二次汇总先保证每条规则至少有一条最高价值证据，再按价值补充其他页，避免文档前部普通命中挤掉后部关键反证或计分材料。网关继续本地兼容模型常见的未转义换行、尾逗号、全角结构符、文本内容块和双重 JSON 编码。

全文范围一致性采用通用的两阶段机制。综合评审开始时，系统仅根据招标文件与已确认规则一次性生成并缓存“项目范围画像”，其中包含项目主体、工作范围、服务对象、核心任务、技术主题、设备/材料、交付物、适用标准、地域及区分度关键词。每个投标文件连续页块的首轮 AI 扫描在规则证据之外，独立返回“项目范围偏离候选”：其类型不预设为地区、项目名或公司名，可以是无关技术路线、设备材料、规范标准、服务对象、成果交付、时间地点或任何其他与范围画像不一致的内容。候选仅是线索，不自行作出废标或不满足结论；模型会给出原文摘录、与范围的关系和优先级。

所有审查规则组都会收到范围画像、偏离候选及对应原页，不再由“地区”“无关项目”等规则关键词决定是否进入复核。因此，后半册出现的无关设备、技术方案或标准不会因本地关键词限额而漏掉；同时，注册地址、联系地址、业绩、公司介绍、合理引用及可选方案不会仅因内容不同被自动认定异常。最终是否构成问题仍由 AI 结合该规则与原文提出建议，并保留给人工审核。

MiniMax M3 的全文扫描、审查和客观评分临时禁用思考，以提高严格 JSON 稳定性；主观方案评分仍保留自适应思考，模型档案的默认设置不会被永久改变。模型若只发生 JSON 语法或转义异常，系统先把当次短响应交给独立、可配置的“结构化结果修复”提示词，不重发数万字投标文件；输出确实达到长度上限时才拆小当前规则组。规则组只漏回部分规则时，仅补评缺失规则，已有结果不重算。以上重试都不把响应正文写入数据库或日志。

任务结果将分别记录 JSON 修复次数、严格紧凑重试次数、规则组拆分次数和缺失规则补评次数；“格式恢复次数”仅作为兼容旧结果的汇总字段，不能再被理解为紧凑重试次数。为降低 MiniMax M3 在长上下文下的 JSON 格式异常，全文扫描的范围偏离候选只保留页码、开放类型、优先级、原文短摘录和与项目范围的直接关系；最终审查与评分仍保留原文证据和判断，但要求理由使用一条短句，不重复叙述已有证据。

最低价、评审价、基准价、价格分或报价得分等必须跨投标人计算的客观规则，在各单份文件证据提取完成后增加一次统一价格评分：将全部投标人的报价相关页与同一规则一起交给 AI，核对口径、币种、最高限价和异常报价后计算建议分，并覆盖单文件阶段无法横向比较而留下的空分。该流程只增加一个按需调用，不引入常驻服务，提示词可从页面右上角查看和修改。

右上角“提示词配置”按维护频率分为三级：最上方“常用业务指令”集中展示文件查重、评审规则和综合评审的通用判断原则，默认展开并推荐优先修改；“业务流程细节”包含规则映射/编译/覆盖审计、全文扫描、审查与评分等带 JSON 输出协议的专项模板，默认折叠并提示必须保留字段和运行时变量；“系统与结构模板”包含模型测试、系统角色、JSON 修复及旧流程兼容模板，默认整体折叠。分组只影响界面展示，所有模板仍可查看、编辑和恢复默认，不改变现有接口或模型调用行为。

综合评审默认自动复用完全相同输入的已完成任务。统一执行指纹覆盖主招标文件、招标附件、投标文件、解析状态、规则集、模型公开配置与全部提示词，不包含正文或密钥；范围画像和全文扫描缓存也绑定该指纹。API 的 `force_rerun=true` 会同时跳过整任务、单份投标文件和全文扫描缓存复用，确保提示词或模型调整后一定重新调用。新增投标文件时，未变化文件仍可按同一完整指纹逐份复用。AI 查重同样复用完全相同输入的完成结果，且其模型请求仅记录压缩证据包的字符数，不保存证据正文。

## 新建项目口令

新建项目必须在页面输入口令。当前部署可在运行环境的 `.env` 中设置 `EVALUATION_WORKBENCH_NEW_PROJECT_PASSWORD`，服务端仅在内存中读取并与请求值作恒定时间比较；不写入数据库、日志、前端代码或仓库。为兼容既有部署，仍支持 `EVALUATION_WORKBENCH_NEW_PROJECT_PASSWORD_HASH`（Werkzeug 密码哈希）；若两者同时存在，明文运行环境配置优先。未配置时服务端拒绝创建项目并返回明确提示。

## 规则来源与确认

AI 从招标文件提取的规则会直接写入当前项目的“待确认规则集”，在规则表中逐条标记为“AI 提取”；人工新增的规则逐条标记为“人工补充”。人工修改 AI 规则的检查语义、评分口径或启用勾选后会标记为“人工锁定”。重新提取时仅刷新普通 AI 规则，人工补充和人工锁定规则会迁移至新草稿，避免重新提取静默删除人工维护结果或重新启用已取消勾选的规则；AI 规则仅是待确认建议，不能绕过人工确认直接用于审查或评分。

## 任务机制

使用 SQLite 任务表与按需启动的单一 Python 工作进程，不引入 Redis、Celery 或常驻任务容器。工作进程按顺序领取任务，队列为空即退出。项目轮询只传输任务摘要，不重复传输大型 `result_json`，详细结果由专门结果接口按需读取。状态为：排队中、运行中、已完成、失败、已取消、上次运行中断。

项目删除仅在没有排队或运行中任务时允许执行；会同时删除项目的 SQLite 记录、原始上传文件、解析缓存、查重结果、审查结果、评分结果和项目报告目录，以回收服务器磁盘空间。删除前必须由用户在页面确认，且不可恢复。

第一阶段实现项目、文件、解析任务、多文件 PDF 查重任务、任务状态和结果矩阵；第二阶段接入规则中心与 AI 审查；第三阶段接入评分、浏览器打印报告与 CSV 导出。XLSX、DOCX 报告会在不引入常驻重型依赖的前提下作为后续扩展。
