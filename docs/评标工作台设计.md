# 评标工作台设计

更新时间：2026-07-19

## 目标与范围

新增独立 Web 模块“评标工作台”，用于单人、低并发的评标辅助。入口为 `/pingbiao`，API 前缀为 `/api/evaluation-workbench`。

模块包含：

- 评标项目、招标文件/附件与多份投标文件管理；
- 多投标文件查重与横向结果汇总；
- 招标规则提取、人工确认，以及资格性、符合性、实质性和废标点审查；
- 客观分的证据提取与程序计分；
- 主观分的证据化辅助审查；
- 浏览器打印与 CSV 评分报告导出；XLSX、DOCX 留作后续兼容性扩展。

明确不包含：投标文件或技术方案编写、AI 生图、OCR、DOC/WPS 转换、本地大模型、多用户协作，以及此前单独列出的上线安全整改。

## 独立边界

- 不修改 `/bijiao`、`/api/compare` 及其响应结构；
- 不影响其他 Blueprint 或 `rokid_glasses_app` 调用的 API；
- 使用独立的 SQLite 数据库 `data/evaluation_workspace.db`；
- 原始文件与解析缓存存储在 `data/evaluation_workspace/<project_id>/`，结构化结果存储在独立 SQLite；打印报告按需生成，不常驻占用磁盘；
- 新功能使用独立 Blueprint、模板、静态资源、服务与任务进程；
- 只在导航页增加“评标工作台”入口。

## 文件与资源限制

- 第一版支持 PDF 和 DOCX；不做扫描件 OCR，未提取到可检索文本的扫描件会明确标记解析失败；
- 单文件上限 100 MB，PDF 解析上限 2000 页，可提取文本上限 200 万字符；
- 每项目默认最多 10 份投标文件；
- 单文件和整批页数、字符数沿用并逐步复用现有 `/bijiao` 的资源边界；
- 文件解析、文件对比和 AI 请求默认并发均为 1；
- 最多保留 3 个排队任务；
- 评标工作台空闲时不保留工作进程、不加载模型、不建立 AI 连接，也不执行定时扫描。

## 工作流

1. 输入新建项目口令后，创建项目并上传一份主招标文件、招标附件和多份投标文件。
2. 按顺序解析文件并按文件哈希缓存结果。
3. 多投标文件顺序两两比较，形成文件关系矩阵和多文件共同线索。
4. 从招标文件提取资格、符合、实质性、废标、客观分和主观分规则。
5. 用户确认、修改和版本化规则。
6. 对每份投标文件分别执行审查、客观分与主观分辅助分析。
7. 汇总横向结果并导出报告。

## 查重结果

查重沿用现有比对模块对 PDF 的高价值规则：完全雷同、近似雷同、已验证招标共同修改、共同错误、敏感实体、元数据辅助线索及招标原文排除。在此之上增加“横向异常线索”层，分别记录线索维度、S3 人工核验等级、可信度、证据状态、反证提示、判定结果和人工处置，不把多个字段压缩成一个不透明的风险分。

固定规则命中后，系统才会将线索压缩为 AI 证据包：投标人对、规则名称、页码、最多三条短原文、相似度/次数及反证提示。AI 按批复核这些证据包，输出“确认线索、疑似线索、倾向排除或证据不足”、风险、置信度和建议核验事项；不发送完整投标文件对，不按文件对逐次调用。没有线索不调用模型，AI 故障时保留本地固定规则结果并标记 AI 判定未完成。查重复用键包含文件哈希、规则/提示版本和模型公开配置，未变化时可复用完成结果。

系统仅生成 `collusion_signal`（串标相关线索），不自动生成或认定 `statutory_collusion_condition`（法定串标情形）。多维命中只提高人工复核优先级，不提高法律定性，不触发废标或扣分。人工可将每条线索标记为待核验、已核实、已排除或需补充证据，并记录核验说明。

当前实际执行正文雷同、共同异常或错误、相同联系方式、相同人员身份信息、招标原文共同改动和相同文档属性六个维度。联系方式和身份证信息在仅限评标使用的横向汇总中完整展示，以便人工核对；元数据只作辅助线索。报价规律、缴费来源、业绩引用、他方主体残留、共同理解偏差和地址关联因缺少可靠结构化来源而明确披露为“未执行”，禁止从正文数字或模型猜测补齐。后续接入结构化报价时，应使用 Decimal 和中位数/MAD/IQR 等稳健统计，小于 4 个有效样本只做描述性展示。

公共模板处理遵循“先排除、再比较”：提供主招标文件时排除招标原文直接复制，并保留共同实质改动；未提供招标文件时明确降低结果解释力度。其他公共行业模板若未获得确认，不得把模板相似直接升级为串标结论。

结果包含：

- 文件关系矩阵：每个文件对的独立线索维度、线索数量和人工复核优先级；
- 正文雷同、目录、元数据、图片、共同错误、共同修改；
- 文件标签、页码、原文对照和风险说明。

图片第一版只做原始图片哈希与位置记录，不做 AI 图像语义识别。

## 审查与评分

招标规则须先由 AI 提取并经用户确认。每条规则保留名称、类型、招标原文、页码、自动检查能力、评分方式和启用状态；规则修改后形成新版本，旧结果保留并标记其使用的规则版本。

资格、符合、实质性和废标审查结果统一使用：满足、不满足、部分满足、未找到证据、无法通过电子文件判断、需人工复核。模型结论与人工最终结论分开保存；页面以中文展示模型结论，并按投标人分组，在每位投标人内按照异常优先、高中低风险排序。每条结果包含招标依据、投标证据、页码、理由、风险级别、置信度、证据完整度及人工确认结果。

工作台采用 AI 优先、异常复核：仅“满足 + 低风险 + 高置信度 + 证据充分”的审查结论自动进入“待批量确认”；不满足、废标相关、证据不足、低置信度、冲突或只能从签字/盖章/图片/线下材料判断的项目必须人工复核。页面提供一键确认非异常结论，不会静默把 AI 输出写成最终结论。

客观分遵循“AI 提取证据、程序计算分数、用户确认最终分数”。当前自动计分仅支持已确认的“满足即满分”规则；高置信且有证据的建议分作为有效分进入批量确认。数量、阈值、分档、累计上限和价格公式保留人工确认，后续只以受限规则结构扩展，禁止执行模型返回的表达式。主观分同样保存 AI 建议分、有效分、人工最终分和调整留痕；低置信或缺少证据的项目进入异常复核。

AI 提取客观分和主观分时必须返回 `scoring.max_score`。若模型遗漏，但招标原文明确包含“总计/最高/最多/满分 X 分”等表述，工作台仅从该原文补齐满分，并标记为“从招标依据识别”；对于数量累计、区间、分档等客观分，默认保持人工评分，不会误作“满足即满分”。待确认规则中的评分满分可直接在规则表内修改保存，无需删除规则重建。

主观分先按同一规则版本、模型档案和 Prompt 版本独立审查每份投标文件，再做横向汇总。客观分与主观分结果均按投标人分组展示。输出建议分数、证据页码、优点、不足、得扣分理由、置信度和人工最终分。

## 模型档案

模型档案是工作台级全局配置，通过页面右上角“模型配置”入口维护，不归属于任何单一评标项目。项目内各任务只选择已配置的模型档案。

模型档案保存显示名、协议、Base URL、模型名、上下文长度、超时、JSON 模式和思考模式。API Key 可以手动录入：系统用工作台独立密钥加密后保存，密钥文件位于被忽略的 `data/` 持久化目录；读取、编辑、模型列表和 API 响应均不返回明文或密文，只显示“已配置”。旧的 API Key 环境变量配置继续兼容。

每个模型档案提供“测试连接”，以极小请求验证 Base URL、模型 ID、API Key 和兼容参数。测试不写入评标项目数据，也不记录 API Key。

可将已启用且已配置 API Key 的模型设为全局默认模型。新建任务和页面初始选择器自动采用该默认模型，用户仍可为某一次任务临时改选；历史任务继续保存实际使用的档案 ID。默认模型不能直接删除，须先选择其他默认模型；若模型正被排队或运行中的任务使用，系统同样拒绝删除。

内置 DeepSeek 预设：

- `deepseek-v4-flash`：规则提取、结构识别、错别字、客观分证据提取；
- `deepseek-v4-pro`：复杂实质性/废标审查、主观分与横向总结；
- Base URL：`https://api.deepseek.com`。

其他模型通过 OpenAI-compatible 适配器接入；每次任务可在已配置档案中选择模型。初始迁移会将已配置的 `deepseek-v4-flash` 设为全局默认模型；以后所有页面与服务端均优先使用管理员明确设置的全局默认模型，再回退至第一个可用模型。

## Token 与上下文优化

原有“逐份审查、客观分、主观分”三个入口完整保留。另提供可选的“综合评审（省 Token）”：同一模型下，每份投标文件只发送一次正文，同时生成审查、客观分和主观分三类结果，并分别写入原有结果表和页面。该入口不改变人工复核、最终评分、报告和 CSV 导出逻辑。

系统为每次模型调用记录任务、阶段、模型、发送字符数、供应商返回的输入/输出/总 Token、缓存命中 Token 和上下文模式；不记录提示词正文、投标文件原文或 API Key。页面仅显示项目汇总。对于不返回 usage 的 OpenAI-compatible 模型，页面显示发送字符数。

独立客观/主观评分会尝试从已解析的页码文本中选择规则相关页，以避免重复传输整份文件。实现不使用向量数据库、常驻索引或本地模型：工作进程内按页执行轻量短语匹配，任务结束后释放内存。只有每条规则均有本地线索且合并页内容至少比原上下文小 30% 时才使用相关页；任何规则未命中、DOCX 无可靠页码或上下文仍过长时，自动退回原有全文前缀。实质性审查保持全文前缀，避免把检索遗漏误作负面结论。

综合评审可勾选“完全相同的输入复用已完成结果”。复用键由文件哈希、解析状态、规则集版本、模型公开配置更新时间和 Prompt 版本组成，不包含正文或密钥；取消勾选可强制重新调用模型。AI 查重同样复用完全相同输入的完成结果，且其模型请求仅记录压缩证据包的字符数，不保存证据正文。

## 新建项目口令

新建项目必须在页面输入口令。当前部署可在运行环境的 `.env` 中设置 `EVALUATION_WORKBENCH_NEW_PROJECT_PASSWORD`，服务端仅在内存中读取并与请求值作恒定时间比较；不写入数据库、日志、前端代码或仓库。为兼容既有部署，仍支持 `EVALUATION_WORKBENCH_NEW_PROJECT_PASSWORD_HASH`（Werkzeug 密码哈希）；若两者同时存在，明文运行环境配置优先。未配置时服务端拒绝创建项目并返回明确提示。

## 规则来源与确认

AI 从招标文件提取的规则会直接写入当前项目的“待确认规则集”，在规则表中逐条标记为“AI 提取”；人工新增的规则逐条标记为“人工补充”。二者可以在同一待确认版本内共同编辑、删除和确认，但 AI 规则仅是待确认建议，不能绕过人工确认直接用于审查或评分。

## 任务机制

使用 SQLite 任务表与按需启动的单一 Python 工作进程，不引入 Redis、Celery 或常驻任务容器。工作进程按顺序领取任务，队列为空即退出。状态为：排队中、运行中、已完成、失败、已取消、上次运行中断。

项目删除仅在没有排队或运行中任务时允许执行；会同时删除项目的 SQLite 记录、原始上传文件、解析缓存、查重结果、审查结果、评分结果和项目报告目录，以回收服务器磁盘空间。删除前必须由用户在页面确认，且不可恢复。

第一阶段实现项目、文件、解析任务、多文件 PDF 查重任务、任务状态和结果矩阵；第二阶段接入规则中心与 AI 审查；第三阶段接入评分、浏览器打印报告与 CSV 导出。XLSX、DOCX 报告会在不引入常驻重型依赖的前提下作为后续扩展。
